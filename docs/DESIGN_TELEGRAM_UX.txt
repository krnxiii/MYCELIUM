================================================================
  F1. VOICE INPUT
================================================================
Backlog: L1.2 | Complexity: ** | Priority: HIGH

Голосовые сообщения → транскрипция → обработка агентом.
Самый естественный способ ввода с телефона.

1.1 STT Provider (pluggable, выбор при установке)
---------------------------------------------------
  Два провайдера на выбор:

    A. Local Whisper (faster-whisper) — бесплатно, приватно
       + Суверенность: аудио не покидает сервер
       + Бесплатно навсегда
       + Качество: модель medium отлично понимает ru/en
       - Скорость: ~10-30s на CPU (VPS без GPU)
       - RAM: +2GB для модели medium
       - Первый запуск: скачивание модели (~1.5GB)
       Реализация: faster-whisper (CTranslate2, оптимизирован для CPU)
       Модель: medium (баланс качество/скорость)
       Docker: отдельный контейнер с предзагруженной моделью

    B. Deepgram Nova-3 — быстрый облачный API
       + Скорость: ~1-2s (real-time)
       + smart_format, punctuate, language auto-detect
       + Прямой upload bytes (без ffmpeg)
       - Внешний API: аудио уходит на сервер Deepgram
       - Нужен аккаунт: deepgram.com → API key
       - Стоимость: $200 free credits, далее ~$0.0043/min
       Реализация: deepgram-sdk (async)

  Выбор при установке (install-vps.sh):
    ┌─────────────────────────────────────────────┐
    │  Voice transcription provider:              │
    │    1) Local Whisper (free, private, slower)  │
    │    2) Deepgram (fast cloud API, needs key)  │
    │    3) Skip (no voice support)               │
    └─────────────────────────────────────────────┘
    → Whisper: ничего дополнительно не нужно
    → Deepgram: запрашивает API key
    → Skip: voice handler не регистрируется

  Config:
    MYCELIUM_TELEGRAM__STT_PROVIDER=whisper-local|deepgram|none
    MYCELIUM_TELEGRAM__STT_API_KEY=...          (только для deepgram)
    MYCELIUM_TELEGRAM__STT_MODEL=medium         (только для whisper)
    MYCELIUM_TELEGRAM__STT_LANGUAGE=auto        (или "ru", "en")

1.2 Pipeline
-------------
  User sends voice message in Telegram
    ↓
  aiogram handler: message.voice detected
    ↓
  bot.get_file() → bot.download_file() → bytes in memory
    ↓
  STTProvider.transcribe(audio_bytes) → text
    ↓
  Отправить транскрипт пользователю: "🎤 <i>{transcript}</i>"
    ↓
  Передать transcript в Dispatcher как обычный текст:
    - Если начинается с "/" → fast mode
    - Иначе → agent full mode

  Ключевое решение: voice = text input.
  Агент получает транскрипт и работает с ним как с обычным текстом.
  Никакой специальной логики обработки голоса — унификация через текст.

1.3 Реализация
---------------
  Новые файлы:
    mycelium/telegram/stt.py — STT provider abstraction + implementations

  Изменения:
    bot.py                   — voice handler
    config.py                — STT settings
    pyproject.toml           — optional deps
    docker-compose.vps.yml   — whisper container (if whisper-local)
    scripts/install-vps.sh   — STT provider selection step

  ~100 LOC новый код.

  ```python
  # stt.py — provider abstraction
  class STTProvider(Protocol):
      async def transcribe(self, audio: bytes) -> str: ...

  class WhisperLocalSTT:
      """Local faster-whisper via HTTP API (Docker container)."""
      async def transcribe(self, audio: bytes) -> str:
          async with httpx.AsyncClient() as client:
              resp = await client.post(
                  f"{self._url}/v1/audio/transcriptions",
                  files={"file": ("voice.ogg", audio, "audio/ogg")},
                  data={"model": self._model, "language": self._language},
              )
              return resp.json()["text"]

  class DeepgramSTT:
      async def transcribe(self, audio: bytes) -> str:
          response = await self.client.listen.asyncio.transcribe_file(
              {"buffer": audio}, model="nova-3",
              language=self._language, punctuate=True, smart_format=True,
          )
          return response.results.channels[0].alternatives[0].transcript

  def create_stt(cfg: TelegramSettings) -> STTProvider | None:
      match cfg.stt_provider:
          case "whisper-local": return WhisperLocalSTT(cfg)
          case "deepgram":      return DeepgramSTT(cfg)
          case _:               return None  # voice disabled

  # bot.py — voice handler
  @router.message(F.voice)
  async def handle_voice(message: Message, dispatcher: Dispatcher, stt: STTProvider):
      file = await message.bot.get_file(message.voice.file_id)
      audio = await message.bot.download_file(file.file_path)
      transcript = await stt.transcribe(audio.read())
      await message.reply(f"🎤 <i>{html.escape(transcript)}</i>", ...)
      # Route transcript through dispatcher as regular text
      channel_msg = ChannelMessage(text=transcript, chat_id=str(message.chat.id))
      async for reply in dispatcher.dispatch(channel_msg):
          ...
  ```

2.2 Media groups (albums)
--------------------------
  Telegram sends album as N separate messages with same media_group_id.

  Buffering strategy (from design doc §6.3):
    - On first photo with media_group_id → start 500ms timer
    - Collect all photos in group during window
    - After timeout → batch process as single signal

  Implementation: MediaGroupBuffer middleware or handler logic.

  ```python
  _album_buffers: dict[str, list[Message]] = {}
  _album_timers: dict[str, asyncio.TimerHandle] = {}

  @router.message(F.photo)
  async def handle_photo(message: Message, ...):
      group_id = message.media_group_id
      if group_id:
          _album_buffers.setdefault(group_id, []).append(message)
          # Reset/set timer
          if group_id in _album_timers:
              _album_timers[group_id].cancel()
          loop = asyncio.get_event_loop()
          _album_timers[group_id] = loop.call_later(
              0.5, lambda: asyncio.create_task(_process_album(group_id, ...))
          )
      else:
          await _process_single_photo(message, ...)
  ```

================================================================
  F4. PROGRESS FEEDBACK (ELAPSED TIME)
================================================================
Backlog: L1.5 | Complexity: * | Priority: MEDIUM

Вместо просто "typing..." → показывать elapsed time в сообщении.
Критично для операций >10s (agent full mode, ingest).

4.1 Текущее состояние
-----------------------
  - TypingKeepAlive: отправляет "typing" action каждые 3s
  - StreamingDelivery: editMessageText для agent output

  Проблема: между отправкой запроса и первым чанком ответа
  может пройти 5-30s. Пользователь не видит прогресса.

4.2 Решение: ProgressIndicator
--------------------------------
  Отдельный компонент поверх StreamingDelivery.

  ```python
  class ProgressIndicator:
      """Show elapsed time until first content arrives."""

      def __init__(self, bot: Bot, chat_id: int):
          self._bot = bot
          self._chat_id = chat_id
          self._msg_id: int | None = None
          self._task: asyncio.Task | None = None
          self._start = time.monotonic()

      async def start(self) -> None:
          msg = await self._bot.send_message(self._chat_id, "⏳ Thinking...")
          self._msg_id = msg.message_id
          self._task = asyncio.create_task(self._tick())

      async def _tick(self) -> None:
          while True:
              await asyncio.sleep(5)
              elapsed = int(time.monotonic() - self._start)
              await self._bot.edit_message_text(
                  f"⏳ Thinking... ({elapsed}s)",
                  chat_id=self._chat_id,
                  message_id=self._msg_id,
              )

      async def stop_and_replace(self, stream: StreamingDelivery) -> None:
          """Cancel timer, delete progress message, hand off to stream."""
          if self._task:
              self._task.cancel()
          if self._msg_id:
              await self._bot.delete_message(self._chat_id, self._msg_id)
  ```

  Альтернатива: переиспользовать progress message как первое
  сообщение StreamingDelivery (edit вместо delete+send).


================================================================
  F5. HTML SANITIZER (3-LAYER)
================================================================
Backlog: L1.6 | Complexity: * | Priority: MEDIUM

Текущее: наивный _strip_html (regex). Не обрабатывает
невалидные теги, не балансирует, не обрезает правильно.

5.1 3-layer pipeline
---------------------
  Layer 1: SANITIZE
    - Оставить только Telegram-safe теги: b, i, code, pre, a, s, u
    - Удалить все остальные теги (не escape, а strip)
    - Экранировать спецсимволы в тексте: & < >

  Layer 2: VALIDATE
    - Проверить баланс открывающих/закрывающих тегов
    - Автозакрыть незакрытые теги
    - При невалидном HTML → fallback to plain text

  Layer 3: TRUNCATE
    - Обрезать до 4096 символов
    - Не разрывать теги при обрезке
    - Закрыть открытые теги после обрезки

  ```python
  ALLOWED_TAGS = {"b", "i", "code", "pre", "a", "s", "u", "blockquote"}

  def sanitize_html(text: str, max_len: int = 4096) -> str:
      """Sanitize → validate → truncate for Telegram."""
      # Layer 1: strip non-allowed tags
      text = _strip_disallowed_tags(text)
      # Layer 2: balance tags
      text = _balance_tags(text)
      # Layer 3: truncate with tag awareness
      if len(text) > max_len:
          text = _truncate_balanced(text, max_len)
      return text
  ```

  ~60 LOC. Заменяет текущий _strip_html в bot.py.

  scheduler = AsyncIOScheduler()

  @scheduler.scheduled_job("cron", hour=21, minute=0)
  async def daily_digest():
      health = await mcp.call_tool("health")
      signals = await mcp.call_tool("get_signals", {"limit": 10})
      metrics = await mcp.call_tool("get_metrics")
      text = format_digest(health, signals, metrics)
      await bot.send_message(owner_chat_id, text, parse_mode="HTML")

================================================================
  USER GUIDE — что умеет бот и как этим пользоваться
================================================================

Ниже — описание каждой фичи простым языком.
Можно использовать как основу для /commands или onboarding.


F1. Голосовые сообщения
------------------------
Что: Записываешь голосовое — бот переводит в текст и обрабатывает.

Зачем: Не всегда удобно печатать, особенно на ходу. Голосом
быстрее — надиктовал мысль, бот сам разберётся что с ней делать.

Как:
  Просто запиши голосовое сообщение в чате с ботом.
  Бот покажет расшифровку и обработает как обычный текст.
  Если это вопрос — ответит из графа.
  Если это мысль — сохранит в граф.
  Ничего дополнительно нажимать не нужно.


F2. Фото и файлы
------------------
Что: Отправляешь фото или документ — бот сохраняет в хранилище
и добавляет в граф знаний.

Зачем: Сфотографировал заметку, схему, страницу книги, чек —
отправил боту, и она уже часть твоей базы знаний. Не нужно
вручную описывать или куда-то складывать.

Как:
  Отправь фото или файл (PDF, текст) в чат.
  Можно добавить подпись — бот использует её как описание.
  Несколько фото подряд (альбом) — бот соберёт в один пакет.
  Бот ответит что сохранено и что извлечено.


F3. Пересланные сообщения
---------------------------
Что: Пересылаешь сообщение из другого чата — бот сохраняет
его вместе с информацией об источнике (кто/откуда).

Зачем: Увидел полезное в чате или канале — переслал боту.
Он запомнит не только текст, но и откуда пришло: от кого,
из какого канала. Потом можно найти: "что мне пересылал Вася"
или "что было из канала про AI".

Как:
  Просто перешли (forward) любое сообщение боту.
  Бот покажет: "Сохранено от [источник]" и обработает содержимое.


F4. Индикатор обработки
-------------------------
Что: Пока бот думает — показывает сколько секунд прошло.

Зачем: Иногда ответ занимает 10-30 секунд (агент ищет по графу,
анализирует). Без индикатора непонятно — бот завис или работает.
Теперь видно: "⏳ Thinking... (15s)" — значит всё ок, ждём.

Как:
  Ничего делать не нужно. Индикатор появляется автоматически
  при долгих операциях и исчезает когда ответ готов.


F5. Корректное форматирование
-------------------------------
Что: Ответы бота правильно форматируются — жирный, курсив,
код — и не ломаются при длинных сообщениях.

Зачем: Раньше при длинных ответах или сложном форматировании
могли появляться обрезанные теги или крякозябры. Теперь бот
аккуратно обрезает длинные сообщения и следит за корректностью.

Как:
  Ничего делать не нужно. Работает автоматически под капотом.


F6. Вечерняя сводка
---------------------
Что: Каждый вечер бот присылает краткий отчёт за день.

Зачем: Не нужно самому спрашивать "что нового". Бот расскажет:
сколько мыслей сохранено, какие нейроны появились, какие знания
начали "выцветать" (давно не вспоминались) и что стоит повторить.

Как:
  Включается в настройках (по умолчанию выключено).
  Приходит в заданное время (по умолчанию 21:00).
  Просто читаешь — если что-то интересно, пишешь боту
  для уточнения.


F7. Еженедельный отчёт
------------------------
Что: Раз в неделю бот присылает глубокий анализ графа знаний.

Зачем: Даёт перспективу: как растёт база знаний, какие темы
доминируют, какие забыты, есть ли противоречия. Помогает
увидеть паттерны, которые не замечаешь день ко дню.

Как:
  Включается в настройках (по умолчанию выключено).
  Приходит раз в неделю (по умолчанию воскресенье, 20:00).
  Содержит: рост за неделю, топ тем, заброшенные области,
  рекомендации.


F8. Улучшенный агент
----------------------
Что: AI-агент лучше понимает контекст твоего графа и точнее
подбирает инструменты для ответа.

Зачем: Раньше агент начинал "с чистого листа" — не знал
что в графе, какие домены, сколько нейронов. Теперь при старте
диалога он получает снимок графа и отвечает точнее с первого
сообщения. Меньше лишних запросов, быстрее ответ.

Как:
  Ничего делать не нужно. Агент автоматически подгружает
  контекст графа при начале новой сессии.
